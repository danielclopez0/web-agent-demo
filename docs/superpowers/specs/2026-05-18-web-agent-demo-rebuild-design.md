# Web Agent Demo — rebuild design

**Date:** 2026-05-18
**Status:** Approved for implementation planning
**Supersedes:** the unstyled, pre-commit prototype transferred from another machine

## Intent

A demo repo that proves AI agents can operate enterprise software through browser control alone — no API integration required. The headline use case is **"point the agent at any site, the agent operates it."** A small bundled sandbox (DemoCorp ERP) gives the README's quickstart a concrete target to demo against on first clone.

The differentiating moment vs every other browser-agent demo: a **browser → local-file → browser** round-trip. The agent navigates the sandbox, triggers a CSV export, captures the file in the project directory, analyzes it locally, generates a chart-rich HTML report, and opens the report in the browser. One prompt, web + filesystem + visualization.

## What the repo contains

Three things, ordered by importance:

1. **`AGENTS.md` + `.devin/skills/`** — the actual product. Tool-agnostic instructions any MCP-capable coding agent can follow (Devin, Claude Code, Cursor, etc.) with a slight Devin lean in worked examples.
2. **DemoCorp sandbox** — three pages, looks "good enough" to feel like an internal corp tool: login → list view → form view (which triggers the CSV download).
3. **One canonical worked example** — the `demo` skill walks the full browser → CSV → analysis → HTML report loop end-to-end against DemoCorp.

## Stack

| Layer | Choice | Reason |
|---|---|---|
| Runtime | Node 22 LTS | Pinned via `.nvmrc` + `"engines"` field. Stops the silent EBADENGINE drift that bit this session. |
| Build | Vite 7.x | Pinned below 8 — Vite 8 + Tailwind v4 combination produced empty utility CSS in diagnostics. |
| UI | React 19 + TypeScript | Boring, well-known, what coding agents scaffold best. |
| Styling | Tailwind v3.4 | The mature stable line. `tailwind.config.js` with explicit `content: [...]`. No module-graph autodetection. |
| Routing | None — plain `<a href>` between pages | Three pages don't need a router. Saves the React Router dep. |
| State | `useState` + `sessionStorage` | Only shared state is the logged-in user. No context, no reducer. |
| Tests | One Playwright smoke test | Verifies login + form submission round-trip. Not 18 tests against a sandbox that's scaffolding. |

## The download-to-project-dir mechanism

The wrinkle that distinguishes this demo:

**App side**: a normal "Export CSV" button on the form/list page that triggers a standard browser download (`Content-Disposition: attachment`). No special protocol — looks identical to any enterprise app's CSV export.

**Agent side**: the agent intercepts the download and saves it to `<project>/exports/<filename>-<timestamp>.csv` instead of `~/Downloads`. Two approaches in priority order:

1. **Preferred — Playwright `download.saveAs()`**: if `@nichochar/playwright-mcp` exposes the download event, the agent listens, calls `saveAs()` with the project-relative path. Looks and behaves like a real user export.
2. **Fallback — direct fetch**: if the MCP doesn't expose downloads, the agent inspects the page, locates the export endpoint, and `curl`s the CSV directly into the project dir. Less impressive narration but rock-solid.

**Implementation order**: verify Playwright MCP support during the first implementation slice. If it works, go with #1. If not, fall back to #2 without redesigning anything else.

## Document structure

Three docs, each does one thing:

- **`README.md`** — quickstart for "I cloned this." Install, run the worked demo, then "point the agent at your own site." ~100 lines.
- **`AGENTS.md`** — auto-loaded by coding agents. Security rules (read-only by default, never enter credentials, ask before writes), the BYO-site workflow, pointer to skills.
- **`.devin/skills/`** — three skills:
  - `demo/SKILL.md` — the worked DemoCorp → CSV → report example (the wow moment).
  - `browse/SKILL.md` — the generic BYO-site flow (navigate any URL, user handles login, agent operates).
  - `analyze/SKILL.md` — "I have a CSV in my project, write me an HTML report" — chart-rich, opened in the default browser.

## The HTML report

Generated by the agent at demo time, not shipped from the repo. The agent picks the visualization approach (inline SVG, Chart.js via CDN, plain HTML tables, etc.) — the `analyze` skill describes the desired qualities (chart-rich, single self-contained file, opens cleanly via `file://`), not a fixed template.

This keeps the repo itself tiny — no chart library dependency, no template engine — and showcases the agent's own judgment as part of the demo.

## Out of scope for this rebuild

- Authentication beyond the hardcoded demo credentials (`john.smith@democorp.example` / `Acme2024!`)
- Real backend / database — all data is in-memory seed data
- Multi-tenant or role-based access
- The architecture-diagram page (cut)
- Mock Jira MCP (cut)
- Comprehensive Playwright test suite (cut to one smoke test)

## Open questions to resolve during implementation

1. **Playwright MCP `download.saveAs()` support** — verify in the first implementation slice. Fall back to curl if unavailable.
2. **CSV schema** — what does the exported CSV contain? Probably purchase orders with vendor, amount, status, cost-center fields. Settle the columns during data-modeling slice; needs to be rich enough that the agent-generated HTML report has interesting things to chart.
3. **Visual fidelity bar** — should DemoCorp look like a 2024 SaaS or a 2008 intranet? Slight preference noted earlier for "enterprise-clunky" since that reinforces the "if you can see it in a browser, an agent can use it" pitch. Confirm during the styling pass.
